# Environment Variables

DeployDash uses [t3-env](https://env.t3.gg) with Zod to validate all environment variables at build time. If a required variable is missing or malformed, the app crashes loudly instead of silently failing in production.

## Usage

Never use `process.env` directly. Always import from `@/core/env`:

```typescript
// correct
import { env } from "@/core/env";
const key = env.STRIPE_SECRET_KEY;

// forbidden — no validation, no type safety
const key = process.env.STRIPE_SECRET_KEY;
```

## Adding a new variable

1. Add it to `src/core/env/index.ts`:

```typescript
server: {
  MY_NEW_SECRET: z.string().min(1),
},
runtimeEnv: {
  MY_NEW_SECRET: process.env.MY_NEW_SECRET,
},
```

2. Add it to `.env.example` with a fake value:

```bash
MY_NEW_SECRET=your_secret_here
```

3. Add the real value to `.env.local`.

## Rules

- Every environment variable must be defined in `src/core/env/index.ts`
- Server variables are never exposed to the client
- Client variables must be prefixed with `NEXT_PUBLIC_`
- `.env.local` is never committed — add to `.gitignore`
- `.env.example` is always committed and always up to date
