Phase 0 — Understand what you're actually building
Research · 3–4 days
Before any code
0.1Run create-t3-app and read every generated file
You need to understand what a CLI-generated project looks like from the inside. Don't just scaffold it — read every single file and understand why it's there. This is your baseline.
Mental model of what your CLI needs to produce
0.2Read Kirimase source code on GitHub
github.com/nicoalbanese/kirimase — read the src/ folder. Specifically: how init writes deploydash.config.json, how add reads it and copies templates, how it modifies existing files. This is your CLI architecture blueprint.
Notes on file writing, config reading, template copying patterns
0.3Read shadcn CLI source code
github.com/shadcn-ui/ui — packages/shadcn/src. This is the gold standard for CLI UX. Understand how it uses clack for prompts, how it resolves paths, how it handles the registry. Your CLI should feel exactly like this.
Understanding of modern CLI UX patterns
0.4Sketch your generated project folder structure
On paper or in a markdown file — decide exactly where every generated file lives. src/lib/auth.ts, src/modules/waitlist/, etc. This is permanent. Every template you write will hardcode these paths. Do not skip this.
ARCHITECTURE.md — your folder structure decision, locked in
0.5Design deploydash.config.json schema
Decide every field this file will ever need. version, auth, db, email, billing, ui, modules[], theme. Add a version field. Treat this like a db schema — changing it later breaks users' projects.
Config schema documented in ARCHITECTURE.md
📄You have ARCHITECTURE.md. You understand what you're generating before generating it.
Phase 1 — Build the CLI engine (no templates yet)
Build · 1 week
Week 1–2
1.1Set up the CLI package
Create a new repo: deploydash-cli. Init as a Node.js + TypeScript package. Configure package.json bin field so npx deploydash works. Set up tsup for building. Test that npx deploydash logs "hello" — that's it.
npx deploydash runs locally via npm link
1.2Build the prompt system with clack
Implement the full init prompt flow using clack: project name → auth provider → db → email → billing → UI lib → which modules. No file writing yet — just get the prompts working and collect the answers into a config object.
npx deploydash init asks all questions, logs the config object
1.3Build the file writing utilities
Write helpers: createFile(path, content), copyTemplate(templatePath, destPath), modifyFile(path, marker, injection), readConfig(), writeConfig(). These are the only tools your entire CLI uses. Get them solid and well-typed.
A utils/fs.ts file with all file operations — tested manually
1.4Implement config read/write + license check
On init: validate license key against your API, write deploydash.config.json to project root. On every subsequent command: read that file first. If it doesn't exist, error clearly. Build this middleware once — every command runs through it.
License validation works. Config persists between commands.
⚙️The CLI engine works. It prompts, validates, reads/writes config, checks license. Zero templates yet — that comes next.
Phase 2 — Build the base template (what init scaffolds)
Build · 2 weeks
Week 3–4
2.1Build the base Next.js template by hand first
Don't template what you haven't built. Create a real Next.js + TypeScript + Tailwind + shadcn project manually. This is the exact project deploydash init will generate. Get it working perfectly before turning it into templates.
A working hand-built Next.js project — your ground truth
2.2Bake in security from the start
Into your hand-built project: t3-env Zod validation for all env vars, security headers in next.config.js (CSP, HSTS, X-Frame-Options), Upstash rate limiting middleware on API routes. Every generated project gets these automatically. Non-negotiable.
securityheaders.com gives your base template an A rating
2.3Convert the hand-built project into CLI templates
Now turn it into templates in your CLI repo. Static files get copied verbatim. Files that vary by stack choice (like a db client that imports Prisma vs Mongoose) become template functions. Follow the folder structure you locked in Phase 0.
npx deploydash init scaffolds a real working Next.js project
🏗️npx deploydash init produces a real, secure, working Next.js project. This is your foundation.
Phase 3 — Adapter pattern + first two providers
Build · 2 weeks
Week 5–6
3.1Write the four adapter interfaces in TypeScript
AuthAdapter, DbAdapter, EmailAdapter, BillingAdapter. These are TypeScript interfaces only — no implementation. Every method your feature modules will ever need must be on these interfaces. We'll design these together when you get here.
src/adapters/types.ts — four interfaces, fully typed, no implementation
3.2Implement Clerk as AuthAdapter
Build Clerk's implementation of AuthAdapter. Every auth operation in every module calls adapter.getUser(), adapter.requireAuth() — never Clerk directly. This is what makes swapping possible. Then wire it into the base template.
Auth works via adapter. Zero direct Clerk imports outside the adapter file.
3.3Implement Supabase + Prisma as DbAdapter
Same pattern. Prisma client wrapped by DbAdapter. RLS policies written for every table by default. Connection pooling configured. Then wire into the base template alongside auth.
DB works via adapter. RLS on every table. Types flow from Prisma to the app.
3.4Implement Resend + Stripe as Email and Billing adapters
Same pattern for the remaining two adapters. Stripe webhook handler gets signature verification baked in — this is where most boilerplates are insecure. Yours won't be.
All four adapters implemented. Full default stack wired together and working.
🔌Full default stack works through adapters. Swap any provider by changing one config value.
Phase 4 — Feature modules
Build · 3 weeks
Week 7–9
4.1Same rule — build each module by hand first
Before templating any module, build it as a real feature in your hand-built project. Waitlist: form, referral logic, admin view, email trigger. Dashboard: authed shell, sidebar, settings. Onboarding: multi-step wizard. Then and only then convert to templates.
Each module working in a real app before becoming a template
4.2Build npx deploydash add [module]
The add command reads deploydash.config.json, copies the right module templates, and modifies existing files using your marker system. Build this for waitlist first — it's the simplest. Then reuse the same pattern for every other module.
npx deploydash add waitlist works end to end
4.3Remaining modules: dashboard, admin, blog, docs, onboarding, marketing, email flows
One by one. Build by hand → convert to template → wire into add command. Resist the urge to do them all at once. Each one teaches you something that makes the next one faster.
All modules installable via npx deploydash add [module]
📦Full module library working. A user can go from zero to a complete SaaS scaffold in one session.
Phase 5 — deploydash.design (the visualizer)
Build · 2 weeks
Week 10–11
5.1Define the preset JSON schema (locked forever)
Before building the UI, define the exact JSON schema a preset produces. Version it. This is the contract between deploydash.design and the CLI. Once a user saves a preset, this schema cannot change.
Preset schema documented and identical in both repos
5.2Build the three preview surfaces
Marketing hero, dashboard shell, waitlist page — all driven by CSS variables. Token changes update every surface in real time. These need to look like real products. Use your own DeployDash modules to build the app itself.
Live preview updates as user changes tokens
5.3Preset save system + npx deploydash apply
Presets stored in Supabase. Each gets a unique ID. The apply command fetches by ID and writes tokens into the project. Requires a valid license key to apply — this is your conversion hook.
Full loop: design → save → apply works end to end
